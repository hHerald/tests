# ТЗ: Отслеживание товаров по номеру поставки (FBO) на Wildberries

**Дата:** 10 июля 2026  
**Версия:** 1.0  
**Статус:** Черновик для согласования

---

## 1. Цель проекта

Создать удобную таблицу (или дашборд), которая решает боль клиента:

- Видеть **сколько товара было принято** по каждой конкретной поставке.
- Понимать **сколько товара продано** в разрезе поставок (насколько это возможно через API).
- Выявлять **"зависшие" остатки** — когда по партии давно нет продаж, но товар ещё числится на складе.

Решение должно быть в виде таблицы, которую клиент может регулярно обновлять.

---

## 2. Анализ доступных данных Wildberries API

### Основные эндпоинты (Блок "Отчёты")

| Задача | Возможность | Основной метод API | Ключевые поля | Комментарий |
|--------|-------------|--------------------|---------------|-------------|
| **Принято по номеру поставки** | Полностью | `GET /api/v1/supplier/incomes` | `incomeID`, `nmId`, `techSize`, `quantity`, `status`, `dateClose`, `warehouseName` | Лучший источник по приёмке |
| **Продано по номеру поставки** | Частично (косвенно) | `GET /api/v5/supplier/reportDetailByPeriod` (или новый `POST /api/finance/v1/sales-reports/detailed`) | `nm_id`, `ts_name`, `quantity`, `doc_type_name`, `gi_id` | Прямой связи "продажа → поставка" **нет** |
| **Текущий остаток + последние продажи** | Да | Комбинация `stocks` + детализация реализации | `quantity`, дата продажи | Позволяет выявлять зависшие товары |

### Вывод по API
Wildberries **не предоставляет** прямой связи между конкретной поставкой и продажами в FBO (товары обезличены на складе).  
Можно сделать только **лучшее возможное приближение**.

## 3. Логика расчётов

### Основные меры

| Мера | Как считать | Поля-источники | Группировка |
|------|-------------|----------------|-------------|
| **Принято (шт.)** | `SUM(quantity)` где `status = "Принято"` | `/incomes` | `incomeID` + `nmId` + `techSize` |
| **Продано (шт.)** | `SUM(quantity)` где `doc_type_name = "Продажа"` | Детализация реализации | `nm_id` + `ts_name` |
| **Текущий остаток** | `quantity` на текущую дату | `/api/v1/supplier/stocks` | `nmId` + `techSize` + склад |
| **Дата последней продажи** | `MAX(дата)` | Детализация реализации / `/sales` | `nm_id` + `ts_name` |
| **Дней без движения** | `Сегодня − Дата последней продажи` | — | — |

### Рекомендуемая группировка
- `incomeID` (номер поставки)
- `nmId` + `sa_name` (артикул)
- `ts_name` / `techSize` (размер)
- `warehouseName` (склад)

## 5. Архитектура решения (рекомендация)

### Вариант A (простой и быстрый)
- **Google Sheets + Apps Script** или **Python (pandas)**
- Ежедневная/по запросу выгрузка данных
- Обработка в скрипте → итоговая таблица

### Вариант B (для регулярного использования)
- Хранилище данных (BigQuery / PostgreSQL)
- ETL-процесс (Airflow / Prefect / Python)
- Дашборд в Power BI / Looker Studio / Tableau

### Частота обновления
- **Оптимально:** 1 раз в день (ночью)
- **Минимум:** 1 раз в 2–3 дня

---

## 6. Ограничения и риски

1. **Нет 100% точной связи** "продажа → конкретная поставка".
2. Метод `/api/v1/supplier/incomes` частично устарел (в документации иногда отмечен как deprecated).
3. Данные по продажам в `/sales` хранятся только ~90 дней. Для долгой истории лучше использовать детализацию отчётов реализации.
4. При больших объёмах данных потребуется пагинация (`rrdid` / `limit`).
