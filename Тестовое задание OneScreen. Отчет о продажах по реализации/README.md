## 🎯 Бизнес-проблема и цель проекта

Клиент работает по схеме **FBO** (Fulfillment by Operator) на Wildberries — регулярно отправляет товары на склады маркетплейса.

**Основные боли клиента:**
- Нет прозрачности по количеству **принятого товара** в разрезе конкретных поставок (`incomeID`).
- Сложно понять, **сколько товара продано** из каждой партии (прямой связи нет).
- Есть подозрение на **"зависшие" остатки** — товар числится на складе, но давно не продаётся, что может приводить к потерям.

**Цель:** Создать удобную, регулярно обновляемую таблицу (или дашборд), которая позволяет:
- Видеть принятое количество по каждой поставке.
- Отслеживать продажи в разрезе поставок (лучшее возможное приближение).
- Выявлять проблемные партии с зависшими остатками.

Решение должно быть в формате таблицы, удобной для регулярного использования клиентом.

## 📊 Анализ доступных данных Wildberries API

Wildberries **не даёт прямой связи** между конкретной поставкой и продажами в FBO (товары обезличены на складе). Поэтому возможно только **лучшее возможное приближение**.

### Ключевые эндпоинты

| Задача                        | Возможность     | Основной метод API                              | Ключевые поля                              | Примечание |
|-------------------------------|-----------------|--------------------------------------------------|--------------------------------------------|----------|
| **Принято по номеру поставки**    | Полностью      | `GET /api/v1/supplier/incomes`                  | `incomeID`, `nmId`, `techSize`, `quantity`, `status`, `dateClose`, `warehouseName` | Лучший источник по приёмке |
| **Продано по номеру поставки**    | Частично (косвенно) | `POST /api/finance/v1/sales-reports/detailed` (или `reportDetailByPeriod`) | `nm_id`, `ts_name`, `quantity`, `doc_type_name`, `gi_id` | Прямой связи нет |
| **Текущий остаток + последние продажи** | Да          | Комбинация `stocks` + детализация реализации    | `quantity`, дата продажи                   | Позволяет выявлять зависшие товары |

**Вывод:** Полноценный трекинг возможен только на уровне артикул + размер. Прямой "sale → incomeID" отсутствует.

## 🧮 Логика расчётов

### Основные метрики

| Метрика                  | Как считать                                      | Поля-источники                     | Группировка                          |
|--------------------------|--------------------------------------------------|------------------------------------|--------------------------------------|
| **Принято (шт.)**        | `SUM(quantity)` где `status = "Принято"`        | `/incomes`                         | `incomeID` + `nmId` + `techSize`     |
| **Продано (шт.)**        | `SUM(quantity)` где `doc_type_name = "Продажа"` | Детализация реализации             | `nm_id` + `ts_name`                  |
| **Текущий остаток**      | `quantity` на текущую дату                       | `/api/v1/supplier/stocks`          | `nmId` + `techSize` + склад          |
| **Дата последней продажи** | `MAX(дата)`                                    | Детализация реализации / `/sales`  | `nm_id` + `ts_name`                  |
| **Дней без движения**    | `Сегодня − Дата последней продажи`               | —                                  | —                                    |

### Рекомендуемая группировка
`incomeID` → `nmId` + `sa_name` (артикул) → `techSize` (размер) → `warehouseName` (склад)

---

## ⚠️ Ограничения и риски

1. **Нет 100% точной связи** "продажа → конкретная поставка" в FBO.
2. Метод `/api/v1/supplier/incomes` частично устарел (в документации иногда отмечен как deprecated).
3. Данные по продажам в некоторых эндпоинтах хранятся ограниченное время (~90 дней). Для долгой истории рекомендуется использовать детальные отчёты реализации.
4. При больших объёмах данных потребуется пагинация (`rrdid` / `limit` / `offset`).
