# Кейс: Анализ ёмкости и доступности слотов курьерской доставки (Ламода)

**Проект от команды аналитики и планирования в операционной дирекции**

Полноценное решение кейса по анализу истории наполнения слотов доставки, формулировке метрик доступности и выявлению проблемных слотов с понятным выводом для операционных менеджеров.

## Описание кейса

В Ламоде работает курьерская доставка. Доступность заказов для клиентов регулируется через **ёмкость слотов доставки**.

Слот доставки определяется:
- Районом (полигоном) — `src_zone_id`
- Интервалом времени доставки — `interval_start_dttm` / `interval_end_dttm`
- Временем отсечки (`interval_cutoff_dttm`) — после него слот становится недоступен для выбора

При оформлении заказа остаточная ёмкость слота уменьшается. Менеджеры могут вручную корректировать ёмкость, пока слот открыт.

**Цель кейса:**
- Проанализировать историю наполнения слотов
- Сформулировать метрики, характеризующие **доступность слотов** для клиентов
- Выявить проблемные слоты
- Предоставить детальную информацию в удобном для операционного менеджера виде

## Данные

Используются два датасета (в ноутбуке — subset-версии для демонстрации):

### `intervals.csv` — задекларированная ёмкость слотов
| Поле                        | Описание |
|----------------------------|----------|
| `slot_id`                  | Идентификатор слота (ключ связи с orders) |
| `interval_start_date`      | Дата доставки |
| `level_2`, `level_1`       | Иерархия расположения полигона |
| `src_zone_id`              | Идентификатор зоны доставки |
| `service_level_code`       | Уровень сервиса (`plus15`, `economy`, `exact60`, `same_day` и др.) |
| `interval_cutoff_dttm`     | Время отсечки приёма заказов |
| `interval_start_dttm` / `interval_end_dttm` | Начало и конец интервала доставки |
| `available_capacity_first` | Первый зафиксированный размер ёмкости |
| `available_capacity_last`  | Последний зафиксированный размер ёмкости |
| `was_expanded`             | 1, если ёмкость расширялась вручную |
| `capacity_delta`           | Изменение ёмкости (`last − first`) |

### `orders.csv` — почасовое наполнение слотов заказами
| Поле                        | Описание |
|----------------------------|----------|
| `slot_id`, `interval_start_date` | Ключ связи со слотом |
| `consumed_capacity_hourly` | Кол-во заказов, оформленных в этот час в слот |
| `consumed_capacity_cum`    | Накопленное потребление к этому часу (нарастающим итогом) |
| `order_hour`               | Час оформления заказа (округлён до часа, МСК) |

**Связь таблиц:** many-to-one (много почасовых записей → один слот).

## Решение

### 1. Подготовка данных
- Объединение таблиц по `slot_id` + `interval_start_date`
- Очистка: удаление слотов без истории ёмкости (`available_capacity_first` / `last` = NaN) — их доля незначительна
- Агрегация заказов до уровня слота

### 2. Сформулированные метрики доступности слотов

| Метрика                        | Формула / Определение | Что показывает |
|--------------------------------|-----------------------|----------------|
| `consumed_final`               | `max(consumed_capacity_cum)` по слоту | Сколько всего заказов было оформлено в слот |
| `active_hours`                 | Кол-во уникальных часов с заказами | Сколько часов слот активно наполнялся |
| `avg_hourly_velocity`          | `consumed_final / active_hours` | Средняя скорость заполнения (заказов в час) |
| `max_hourly_velocity`          | Максимум `consumed_capacity_hourly` | Пиковая скорость заполнения |
| `capacity_utilization`         | `consumed_final / available_capacity_first` | Доля использованной начальной ёмкости |
| `sold_out_time`                | Первый `order_hour`, где `consumed_capacity_cum >= available_capacity_first` | Момент исчерпания начальной ёмкости |
| `lead_to_cutoff_hours`         | `(interval_cutoff_dttm − sold_out_time).total_hours` | За сколько часов до cutoff слот закончился (NaN = не закончился) |

**Порог "высокой скорости"** — 95-й перцентиль распределения `avg_hourly_velocity`.

### 3. Классификация проблем (функция `classify_problem`)

Слот может иметь **одну или несколько** проблем одновременно:

| Проблема                        | Условие | Интерпретация для менеджера |
|--------------------------------|---------|-----------------------------|
| **Переполнение**               | `capacity_utilization > 1` | Слот принял больше заказов, чем было заявлено изначально |
| **Расширение ёмкости**         | `was_expanded == 1` | Менеджер вручную увеличивал ёмкость (сигнал о нехватке) |
| **Закончился до cutoff**       | `lead_to_cutoff_hours > 0` | Слот исчерпался **до** времени отсечки — клиенты не могли оформить заказы в последние часы |
| **Высокая скорость заполнения**| `avg_hourly_velocity > 95% percentile` | Слот заполняется аномально быстро (риск раннего исчерпания) |
| **Недозагрузка**               | `capacity_utilization < 0.5` | Использовано менее половины ёмкости — возможно, слот слишком "широкий" или не востребован |
| **Без проблем**                | Нет ни одного условия выше | Слот работает в штатном режиме |

### 4. Итоговая таблица для операционного менеджера

На выходе формируется таблица со всеми ключевыми полями + колонка `problem_type`:

```text
slot_id | interval_start_date | src_zone_id | service_level_code | 
interval_start_dttm | interval_end_dttm | interval_cutoff_dttm |
available_capacity_first | available_capacity_last | was_expanded | capacity_delta |
consumed_final | avg_hourly_velocity | max_hourly_velocity | capacity_utilization |
sold_out_time | lead_to_cutoff_hours | problem_type
```

**Пример проблемных слотов** (из решения):
- Слоты с `Переполнение; Закончился до cutoff`
- Слоты с `Высокая скорость заполнения`
- Слоты с `Недозагрузка`
- Слоты, закончившиеся за 1–26 часов до cutoff

## Ключевые insights из анализа

- Значительная часть слотов имеет проблемы с доступностью (переполнение или раннее исчерпание).
- Высокая скорость заполнения часто приводит к тому, что слот заканчивается задолго до cutoff.
- Недозагрузка (< 50%) встречается у многих слотов — потенциал для оптимизации размеров ёмкости.
- Ручные расширения ёмкости (`was_expanded`) — явный сигнал, что изначальная ёмкость была недостаточной.

## Как использовать результат

Операционный менеджер может:
1. Фильтровать таблицу по `problem_type` (например, только "Закончился до cutoff")
2. Смотреть на `lead_to_cutoff_hours` — приоритет для слотов, которые заканчиваются сильно заранее
3. Анализировать `src_zone_id` + `service_level_code` — выявлять проблемные зоны/сервисы
4. Принимать решения:
   - Увеличивать начальную ёмкость для часто переполняющихся слотов
   - Уменьшать ёмкость для chronically недозагруженных
   - Менять время cutoff или интервалы
